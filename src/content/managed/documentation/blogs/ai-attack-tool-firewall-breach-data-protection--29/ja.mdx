# ファイアウォールが突破された後、自社のデータは安全か。"CyberStrikeAI"事件から学ぶ、多層防御の考え方

## 1. 結論：AI攻撃の大衆化が経営リスクを一変させる

2026年初頭、AIを搭載したオープンソースの攻撃自動化プラットフォームが、55カ国のFortiGateファイアウォール600台以上を突破する事件が発生しました。最も深刻な事実は、この攻撃を実行した脅威アクターが「技術的に未熟」と評価されている点です。AIが攻撃のハードルを劇的に下げ、従来は高度なスキルを要した大規模攻撃を誰でも実行できる時代が到来しました。

経営者が直視すべき問いは明確です。**ファイアウォールが突破された後、自社のデータは安全か。**

## 2. 事件の全貌──何が起きたのか

### 事件概要

2026年1月11日から2月18日にかけて、600台以上のFortiGateファイアウォールが侵害されました。[AWSのCISO CJ Moses氏がこの活動を報告](https://aws.amazon.com/blogs/security/ai-augmented-threat-actor-accesses-fortigate-devices-at-scale/)し、その後サイバーセキュリティ企業[Team Cymruの調査](https://www.team-cymru.com/post/tracking-cyberstrikeai-usage)により、攻撃に「CyberStrikeAI」と呼ばれるAIネイティブな攻撃自動化プラットフォームが使用されていたことが判明しました。

### 主要ファクト

- **被害規模：** 55カ国、600台以上のFortiGate装置
- **攻撃手法：** 既知の脆弱性の悪用ではなく、**露出した管理ポートと弱い認証情報**を狙った攻撃
- **攻撃者の特徴：** ロシア語話者の金銭目的の脅威アクター（AWSの評価）
- **AIの役割：** 攻撃の計画立案、スクリプト生成、運用全体の支援にClaudeやDeepSeek等の商用LLMを使用
- **プラットフォーム開発者：** 中国を拠点とする開発者。[Team Cymruの調査](https://www.team-cymru.com/post/tracking-cyberstrikeai-usage)により中国政府系組織との関連が指摘されている
- **運用インフラ：** 2026年1月20日から2月26日の間に21台のサーバーで稼働を確認（中国・シンガポール・香港に集中）

注目すべきは、**攻撃プラットフォームの開発者（中国拠点）と、実際に攻撃を実行した脅威アクター（ロシア語話者）は別の人物**であるという点です。オープンソースで公開されたツールが国境を越えて利用される現実を象徴しています。

### 開発者の背景と懸念

[Team Cymruの調査](https://www.team-cymru.com/post/tracking-cyberstrikeai-usage)によると、開発者はCNNVD（中国国家情報セキュリティ脆弱性データベース）のLevel 2貢献賞を受賞した経歴がありますが、その後GitHubプロファイルから当該記述を削除しています。また、中国国家安全部（MSS）との関連が指摘されるセキュリティ企業Knownsec 404との交流も確認されています。

[CSO Online](https://www.csoonline.com/article/4140221/ai-powered-attack-kits-go-open-source-and-cyberstrikeai-may-be-just-the-beginning.html)の報道によれば、Beauceron SecurityのDavid Shipley氏はこのツールの公開について次のように警告しています。

> 「この洗練度と実害を与える能力を持つツールをオープンソースで公開するのは無責任だ。倫理的ハッカーやセキュリティ研究者が責任を持って使える従来のツールとは全く次元が異なる」

## 3. なぜ「初心者でも大規模攻撃」が可能になったのか

このプラットフォームが従来の攻撃ツールと根本的に異なるのは、**AIによるエンドツーエンドの攻撃自動化**を実現した点です。

### 主な特徴

- **100以上のセキュリティツール統合：** nmap、sqlmap、nuclei、metasploit等を一元管理
- **AIオーケストレーション：** 自然言語の指示からAIが攻撃チェーンを自動構築・実行
- **MCPプロトコル対応：** Model Context Protocolにより、AIエージェントが外部ツールを直接呼び出し
- **役割ベーステスト：** 侵入テスト、CTF、Webアプリスキャンなど12の定義済みロール
- **攻撃チェーン可視化：** 段階的なリプレイとリスクスコアリング

| 項目 | 従来の攻撃ツール | AI攻撃自動化プラットフォーム |
|---|---|---|
| 技術的ハードル | 高い（専門知識必要） | 低い（自然言語で指示可能） |
| 自動化レベル | 個別ツールごとに手動操作 | エンドツーエンド自動化 |
| 攻撃規模 | 小から中規模 | 大規模マススキャニング |
| コスト | ツール習熟に長期間 | AI APIキーのみで即実行 |
| 入手方法 | クローズドコミュニティ | オープンソースで公開 |

[Dark Reading](https://www.darkreading.com/threat-intelligence/600-fortigate-devices-hacked-ai-amateur)の報道によれば、AWSは今回の攻撃者がAIの支援により「以前であれば大規模な熟練チームを要した規模」の攻撃を単独で実行したと評価しています。Beauceron SecurityのShipley氏はこの変化を「マスケット銃からAK-47への進化に匹敵する」と表現しています。

## 4. ファイアウォールが突破された後に何が起きるのか

今回の事件で見過ごせないのは、ファイアウォールを突破した後の攻撃者の行動です。

### 攻撃後の行動パターン

1. **認証情報の窃取：** オープンソースツールでNTLMパスワードハッシュを抽出し、ドメイン全体の認証情報データベースを取得
2. **横移動（ラテラルムーブメント）：** Pass-the-Hash / Pass-the-Ticket攻撃で内部ネットワークを横断
3. **バックアップ破壊：** [SecurityWeek](https://www.securityweek.com/hundreds-of-fortigate-firewalls-hacked-in-ai-powered-attacks-aws/)の報道によれば、Veeam Backup and Replicationサーバーを標的にし、復旧手段を破壊（ランサムウェア展開の前段階）

ここで経営者が認識すべき現実があります。

> **ファイアウォールは「家の扉」にすぎない。扉を破られた後に守るべきは、金庫（データ）そのものである。**

しかし、多くの企業ではアプリケーションサーバー（WAS）からデータベースへの接続において、きめ細かいアクセス制御が適用されていません。サービスアカウント1つで全テーブルにアクセス可能な構成が一般的であり、ファイアウォール突破後に内部ネットワークへ到達された場合、データベースが事実上無防備になり得るのです。

## 5. 多層防御の実践フレームワーク

AI攻撃時代においては、単一の防御層に依存する戦略は破綻します。必要なのは**多層防御（Defense-in-Depth）** の考え方です。

### 3つの防御レイヤー

**レイヤー1：ネットワーク境界防御**
- ファイアウォール・IPS/IDSによる侵入検知
- 管理ポートの外部露出の遮断
- パッチ管理とファームウェア更新の徹底
- AI駆動のネットワークトラフィック分析

**レイヤー2：アプリケーション・認証防御**
- 多要素認証（MFA）の必須化
- APIキー・認証情報の定期ローテーション
- WAF（Web Application Firewall）による入力検証
- ゼロトラストアーキテクチャの実装

**レイヤー3：データ防御（最後の砦）**
- データベースアクセス制御（スキーマ・テーブル・行レベル）
- 動的データマスキング（機密情報の自動秘匿化）
- クエリ監査ログの全量記録
- 異常クエリパターンの検知とアラート
- Rate Limiting（大量データ抽出の物理的防止）

### 防御の本質

今回の事件が[Team Cymruの報告](https://www.team-cymru.com/post/tracking-cyberstrikeai-usage)で明らかにしたように、攻撃者は「脆弱性」を利用したのではなく、**露出した管理ポートと弱い認証情報**を突きました。[Ciscoの「State of AI Security 2026」レポート](https://www.itmedia.co.jp/enterprise/articles/2602/27/news066.html)によれば、2025年は「AIベースの実験」の時代でしたが、2026年は「AI駆動の攻撃」の時代が始まったと評価されています。

つまり、**基本的なセキュリティ衛生（パッチ適用、強力な認証、不要なポートの遮断）** が、AI攻撃への最も効果的な防御であるという事実は変わりません。

## 6. 防御アプローチ比較

| 防御カテゴリ | ファイアウォール / IPS | SIEM / SOC | データベースアクセス制御 |
|---|---|---|---|
| 防御対象 | ネットワーク境界 | ログ・イベント | データそのもの |
| 検知タイミング | 侵入時 | 侵入後（ログ分析） | クエリ実行時（リアルタイム） |
| AI攻撃への対応 | 既知パターンのみ | 大量ログの分析遅延 | クエリ単位で制御可能 |
| 内部脅威への対応 | 対応困難 | 検知は可能だが遅延 | 行レベルで制御可能 |
| データ保護 | 間接的 | 間接的 | 直接的（マスキング / フィルタリング） |
| 運用負荷 | 中程度 | 高い（専門人材要） | ポリシーベースで自動化可能 |

3つのアプローチは代替関係ではなく**補完関係**です。ファイアウォールが「誰を入れるか」を管理し、SIEMが「何が起きたか」を記録し、データベースアクセス制御が「何にアクセスできるか」を制御します。

## 7. 経営上の示唆と次のアクション

### 重要ポイント

- AI攻撃ツールのオープンソース化により、**攻撃の技術的ハードルが事実上消滅**した
- ファイアウォールだけの防御戦略は、すでに破綻している
- データベースレベルの防御は「あると良い」から「なくてはならない」に変わった
- WAS→DB間のアクセス制御の不在は、多くの企業が抱える構造的リスクである

### 今すぐ（今週中に）

- 外部に露出した管理ポートの棚卸しと遮断
- FortiGate装置のファームウェア最新化の確認
- デフォルトパスワード・弱い認証情報の一斉変更
- バックアップインフラのオフライン保管状況の確認

### 短期（1ヶ月以内に）

- 多要素認証の全社展開状況の点検
- データベースアクセス制御ポリシーの見直し
- クエリ監査ログの取得・保管体制の確認
- インシデント対応計画へのAI攻撃シナリオ追加

### 中期（四半期以内に）

- ゼロトラストアーキテクチャのロードマップ策定
- ネットワーク防御とデータ防御の統合戦略の構築
- 従業員向けセキュリティ教育の更新（AI脅威を反映）

## 8. おわりに：AIの脅威には、AIの防御で立ち向かう

CyberStrikeAI事件は、サイバーセキュリティのパラダイムが完全に移行したことを示す歴史的な転換点です。これまで国家を背後にもつ高度なハッカー集団しか実行できなかった複雑な攻撃チェーンが、今や誰でもダウンロード可能なツールに組み込まれました。

しかし、絶望する必要はありません。攻撃者がAIを自動化の武器として使うのであれば、防御側もまたAIを活用して検知と制御を自動化すべきです。境界防御が破られることを前提とした上で、最も重要な資産である「データベース」に強固なアクセス制御と監査の仕組みを組み込むこと。これが、AI主導の脅威時代における最も現実的かつ強力な経営戦略となるでしょう。
